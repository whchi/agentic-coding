# Product Discovery Reference

Use this reference when the PRD request is still early-stage, ambiguous, or product-heavy.

It is a support document, not a replacement template. Pull in only the parts that improve the current PRD.

## What This Reference Is Good For

- sharpening product vision and positioning
- identifying representative personas and pain points
- mapping the user journey and major touchpoints
- outlining a high-level feature map
- surfacing technical risks and areas that need validation

## Product Discovery Prompts

### 1. Product Vision And Market Positioning

- What is the core value of this feature or product?
- Which users benefit most from it?
- What gap, friction, or unmet need does it address?
- Why is this worth doing now?

Keep this short and meaningful. Only include market positioning when it is relevant to the user's context.

### 2. User Personas And Pain Points

- Who are the primary users?
- What are they trying to accomplish?
- What frustrates them today?
- Which of those pain points does this work actually solve?

Use representative personas only when they help clarify decisions. Do not invent detailed backstories without evidence.

### 3. User Journey And Feature Map

- What does the end-to-end workflow look like today?
- Where are the key touchpoints and decision points?
- Which moments fail, stall, or create confusion?
- What high-level capabilities are needed at each stage?

Use this to improve the PRD's problem, solution, and user story quality.

### 4. Reliability And Technical Considerations

- What technical constraints or dependencies are already visible?
- What architecture, scalability, or reliability risks might matter?
- Which unknowns need validation before implementation begins?

If information is missing, say so explicitly. Unknowns are better than false certainty.
